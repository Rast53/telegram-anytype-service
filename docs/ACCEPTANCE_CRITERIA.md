# Acceptance Criteria (MVP)

## Функциональные

1. Сообщение с планом на сегодня/дату создает или обновляет страницу плана в Anytype.
2. Сообщение без четкой инструкции записывается в Inbox (страница + при наличии `ANYTYPE_INBOX_COLLECTION_ID` — добавление в коллекцию Inbox DB 2Brain).
3. Запрос планов на дату возвращает список пунктов, если план существует.
4. Запрос Inbox возвращает последние записи (из коллекции Inbox DB при заданном `ANYTYPE_INBOX_COLLECTION_ID`, иначе по поиску по префиксу заголовка).
5. Пользователь получает статус о приеме (`task_id`) и итог выполнения.

## Нефункциональные

1. Повторный Telegram `update_id` не создает дубликаты операций.
2. Ошибки MCP/LLM обрабатываются с retry и возвращают человекочитаемый ответ.
3. Сервис имеет `health` и `ready` endpoints.
4. Секреты не выводятся в логах.
5. Понятные запросы обрабатываются rule-engine без LLM вызова.
6. LLM используется только как fallback-классификатор при неочевидных формулировках.

## Тестовые сценарии (unit)

- `tests/dayPlanParser.test.ts` — парсинг планов и дат.
- `tests/intentRouter.test.ts` — rule-first routing и fallback.
- `tests/commandExecutor.test.ts` — форматирование ответа по плану.
