# GitHub Issues + Projects + PR workflow

Как вести задачи, коммиты и код в репозитории так, чтобы агент и человек могли работать по одной схеме: задача → ветка → коммит с номером → PR → мерж.

## 0. GitHub CLI: установка и создание задач агентом

**Установка (один раз):**
```powershell
winget install --id GitHub.cli --source winget
```
При ошибке msstore/certificate используй `--source winget`. Затем закрой и открой терминал (или обнови PATH: `$env:Path = [System.Environment]::GetEnvironmentVariable("Path","Machine") + ";" + [System.Environment]::GetEnvironmentVariable("Path","User")`).

**Вход в GitHub (один раз):** `gh auth login` → GitHub.com, HTTPS, аутентификация (scope `repo`).

**Создание issue из терминала:** из корня репо: `gh issue create --title "Название" --body "Описание."`

## Правила для агента: когда создавать задачу и как коммитить

**Когда создавать задачу (issue):**
- По явной просьбе пользователя («создай задачу», «заведи issue на …»).
- Перед началом нетривиальной работы по формулировке пользователя, если номер issue не указан — создать issue через `gh issue create`, затем ветку `issue-N-short-name` и коммиты с `fixes #N`.
- Не создавать issue на каждую мелкую правку; для точечных изменений — коммит в текущую ветку или в `master` по договорённости с пользователем.

**Как создавать задачу:**
- Из корня репо: `gh issue create --title "Краткое название" --body "Описание и при необходимости чек-лист приёмки."`
- Если `gh` не установлен или не авторизован — сформулировать задачу текстом и попросить пользователя создать issue вручную в GitHub.

**Когда коммитить:**
- Логическими кусками: одна фича, один фикс или один рефакторинг — один или несколько атомарных коммитов.
- Сообщение: краткий заголовок (feat/fix/docs/...), при необходимости тело; если работа по issue — в конце заголовка или в теле указать `fixes #N` (закроет issue при мерже PR) или `refs #N` (только ссылка).

**Ветки и PR:**
- Работа по задаче: ветка `issue-N-short-name` от актуального `master`; в коммитах писать `fixes #N`; пуш, затем PR с `fixes #N` в описании.
- Полный чек-лист: раздел 8 ниже.

## 1. Заводим задачу (Issue)

- **Вариант A:** GitHub → Issues → New issue (или шаблон Bug report / Feature request).
- **Вариант B:** из терминала/агентом: `gh issue create --title "..." --body "..."` (раздел 0 и правила для агента выше).

Пишем название и описание, при необходимости — чек-лист приёмки. Issue получает номер, например `#3`.

Использование:
- Ты создаёшь issue вручную и даёшь агенту: «сделай по #3».
- Агент может создать issue сам через `gh issue create` и дальше работать по его номеру.

## 2. Ветка от задачи

- От `master` создаётся ветка. Рекомендуемый формат: `issue-3-short-name` или `3-short-name`.
- В Cursor/терминале:
  ```bash
  git fetch origin
  git checkout -b issue-3-setup-anytype-nas origin/master
  ```

## 3. Коммиты со ссылкой на issue

В сообщении коммита или в описании PR пишем:
- `fixes #3` или `closes #3` — тогда при мерже PR в `master` issue автоматически закроется.
- `refs #3`, `see #3` — просто ссылка, без автозакрытия.

Пример:
```
feat(scripts): add setup-anytype-nas bootstrap script

Ref: docs/NAS_ANYTYPE_BOOTSTRAP.md
fixes #3
```

## 4. Pull Request

- Пушим ветку: `git push -u origin issue-3-setup-anytype-nas`.
- В GitHub: **Pull requests → New pull request**, база `master`, compare — наша ветка.
- В описании PR указываем `fixes #3` (или `closes #3`), чтобы при мерже закрылся issue.
- После ревью (или сразу, если один разработчик) — **Merge pull request**.

## 5. Projects (канбан, опционально)

- **GitHub → Projects → New project** → выбрать шаблон (Board или Table).
- В настройках проекта включить связь с репозиторием и полем **Status** (или свои поля).
- В настройках колонок можно привязать статус к issue: например, **Todo** / **In Progress** / **Done**.
- Когда создаётся issue, его можно добавить на доску; при закрытии issue карточка переходит в Done (если настроено).

Так ты в GUI видишь задачи, а агент может ориентироваться по номерам issue в коммитах и PR.

## 6. Как с этим работать тебе

| Действие | Где |
|----------|-----|
| Завести задачу | Issues → New issue (или шаблон Bug/Feature) |
| Посмотреть открытые задачи | Issues (фильтр open), при наличии — Projects |
| Взять задачу в работу | Создать ветку `issue-N-title`, в коммитах писать `fixes #N` |
| Смержить и закрыть задачу | PR в `master` с `fixes #N` → Merge → issue закроется |
| Найти коммиты по задаче | В issue внизу видны связанные коммиты и PR |

## 7. Как с этим работать агенту

- Следовать **правилам для агента** (раздел выше): когда создавать задачу, когда коммитить, формат веток и `fixes #N`.
- По задаче пользователя («сделай #5» или «реализуй …») — ветка `issue-5-short-name`, коммиты с `fixes #5`, пуш, предложить открыть PR с `fixes #5` в описании.
- Для трекинга: в начале ответа «Working on #5», в конце — «Ready for PR, branch `issue-5-...`, fixes #5».

## 8. Краткий чек-лист на задачу

1. Создать issue (или взять существующий).
2. `git checkout -b issue-N-korotkoe-imya origin/master`.
3. Разработка, коммиты с `fixes #N` в сообщении.
4. `git push -u origin issue-N-korotkoe-imya`.
5. Создать PR в `master`, в описании — `fixes #N`.
6. После мержа — issue закроется, ветку можно удалить.
